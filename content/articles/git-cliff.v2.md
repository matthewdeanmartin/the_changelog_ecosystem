## What I actually ran

This is a second-pass review grounded in *running* git-cliff, not reading its docs. The
reproducible experiment lives in [`examples/rust/git-cliff/`](https://github.com/matthewdeanmartin/the_changelog_ecosystem/tree/main/examples/rust/git-cliff).

- **Base image:** `debian:bookworm-slim` (runtime); `rust:1.87-slim` (build stage for the tipcalc fixture app)
- **Tool version:** `git-cliff 2.13.1` (pre-built musl binary from GitHub Releases — no `cargo install`)
- **Fixture:** a trivial all-constants "restaurant tip calculator" CLI written in Rust
- **Life cycle, in an isolated in-container git repo:**
  1. v1.0.0 code committed and tagged — no changelog yet.
  2. `git-cliff --config cliff.toml --output CHANGELOG.md` — first changelog generated.
  3. Implement even-split feature; `--unreleased` preview of pending changes.
  4. Tag v2.0.0, regenerate full changelog; repeat for v3.0.0.
  5. Bonus: `git-cliff --latest` for just the last release; `git-cliff --init` to show default config.

## Real output

CHANGELOG.md after the full three-version run:

```markdown
# Changelog

All notable changes to this project will be documented in this file.

## [3.0.0] - 2026-06-02

### Features

- Split the bill unevenly by weight
## [2.0.0] - 2026-06-02

### Documentation

- Add changelog for 1.0.0

### Features

- Split the bill evenly among diners
## [1.0.0] - 2026-06-02

### Features

- Compute tip for a single bill
```

The `--unreleased` preview mid-run (before tagging v2.0.0):

```markdown
## [Unreleased]

### Documentation

- Add changelog for 1.0.0

### Features

- Split the bill evenly among diners
```

The `--latest` flag, run after tagging v3.0.0, produced only the most recent release — useful for pasting into a GitHub Release body.

The default `cliff.toml` generated by `--init` is much more elaborate than the minimal config used in the experiment: it includes emoji group prefixes (`🚀 Features`, `🐛 Bug Fixes`), HTML-comment sort keys, `commit_preprocessors`, security body matchers, and a `<!-- 10 -->💼 Other` catch-all group.

## Pros (observed)

**Works with zero registry round-trips.** Downloading the pre-built musl binary took a few seconds. The binary was self-contained — no shared library dependencies, no system Rust toolchain needed at runtime.

**Tera templates are genuinely powerful.** The minimal `cliff.toml` in the experiment uses `{% for group, commits in commits | group_by(attribute="group") %}` and `| trim_start_matches(pat="v")`. It reads like actual Jinja2 and works as documented.

**Incremental commands compose well.** `--unreleased` showed what would appear in the *next* release without overwriting the file. `--latest` isolated just the last tag. `--output` wrote in place. These flags cover the major CI use cases without requiring separate tools.

**`--init` is a useful scaffolding entry point.** The generated `cliff.toml` is verbose but heavily commented, making it a good starting point rather than requiring you to read the full docs before writing config.

**filter_unconventional = false works as a safety net.** The experiment's `cliff.toml` set this, which meant the `docs: add changelog` commit appeared under a "Documentation" group rather than being silently dropped. The original article noted this flag but didn't show the difference live.

## Cons / pain points (observed)

**Blank lines between release sections are missing.** The generated CHANGELOG.md has no blank line between `## [2.0.0]` and the preceding `## [3.0.0]` block — the sections run together visually. This is a template whitespace issue in the `body` Tera template; you need careful `\n` handling at the end of the body block or a `postprocessors` regex to fix it.

**The `docs:` commit leaks into the changelog.** A `docs: add changelog` commit that is itself a maintenance commit appeared as a real "Documentation" section entry. This is not a bug — the config intentionally includes `docs:` — but it means you need to either skip `docs:` commits or be disciplined about what gets committed. The default `--init` config skips `chore(release):` commits but still includes general `chore:`.

**`--init` output goes to stdout mixed with log lines.** Running `git-cliff --init` in the experiment produced colored ANSI escape codes (`[32;1mINFO[0m`) mixed into the stdout. In a transcript this looks messy; in CI you'd want `--no-exec-path-rewrite` or filtering.

**Timestamp precision.** All three releases show `2026-06-02` because they were all created in the same container run. In real projects this is fine, but it means the changelog does not distinguish intra-day releases.

## Docs vs. reality

The original `git-cliff.md` article described the tool accurately: Tera templates, `--unreleased`, `--latest`, `--init`, and Conventional Commits support all work exactly as documented.

Two things the original article undersold:
1. **Template whitespace discipline.** The article says templates are "flexible" without warning that missing newlines produce cramped output. The hands-on run made this immediately visible.
2. **The default config is opinionated about emoji and sort keys.** The article shows a minimal config example; the real `--init` output is much more prescriptive, which is useful to know before you start editing.

The article noted that git-cliff "can fail CI when commits do not match configured rules" — the `fail_on_unmatched_commit` flag. This is real, but the default is `false` (do not fail), so new users won't hit it unexpectedly.

## Revised verdict

**Keep as a top recommendation, but note the whitespace gotcha.** git-cliff is the most capable and most actively maintained Rust changelog generator. The pre-built binary means zero Rust toolchain dependency at runtime, and the Tera template engine is expressive enough to cover every format we tested elsewhere (KAC, plain markdown, release-notes-only). The blank-line-between-sections issue is a one-line template fix, not a fundamental problem.

For Rust projects: use git-cliff as the changelog engine, call it from `cargo-release` or `release-plz` via hook, and pin a specific version in CI to get reproducible output.
