## Pros (observed)
- Output is genuinely human-facing Markdown with stable headings; no raw git log.
- `--draft` preview is a real, useful feature for PR-time release-note review.
- `towncrier check --compare-with <ref>` is a working CI gate; found the fragment.
- `--date` makes builds deterministic/reproducible — good for testing and CI.
- Newest-first prepend is correct and needs no manual ordering.

## Cons / pain points (observed)
- **Fragments must be git-committed BEFORE `towncrier build`.** towncrier removes
  consumed fragments with `git rm`; if they're untracked you get a noisy
  `fatal: No pathspec was given. Which files should I remove?` mid-build. The build
  still succeeds, but the error looks alarming and would fail a strict CI step.
- **Empty newsfragments/ disappears.** After a build the dir is emptied and git does
  not track empty dirs, so the next stage must `mkdir -p newsfragments` again. A
  `.gitkeep` would avoid this in a real project.
- **No version inference here.** We passed `--version` explicitly every time; towncrier
  can read it from package metadata but that needs the package installed, which is extra
  setup the minimal fixture skips.
- **App bytecode polluted `towncrier check`'s file list** (`__pycache__/*.pyc` showed up)
  until an app `.gitignore` was added — a reminder that check inspects the whole diff.

## Docs vs. reality
- The original article's claim that output is "intentionally human-facing, not a raw
  commit log" — confirmed exactly.
- The article describes `towncrier check` as a CI gate — confirmed it works against a
  base ref.
- What the article (and most docs) undersell: the **git-coupling footgun** — fragments
  must be tracked before build, and the empty-dir disappearance. These only surface when
  you actually run the life cycle, which is the whole point of this experiment.

## Revised verdict
Keep "Recommended". The hands-on run confirms the strengths and adds concrete setup
guidance (commit fragments first; keep newsfragments/ with a .gitkeep). No downgrade.

## Cross-platform note (host = Windows / Git Bash)
- `make run` works: GnuWin32 make resolves `$(CURDIR)` to a Windows path, so the
  `-v` bind mount maps correctly and artifacts return to `out/`.
- Running `docker run -v "/c/.../out:/work/out"` BY HAND from Git Bash silently fails to
  return artifacts — MSYS mangles the paths. If you must run docker by hand on Windows,
  use a Windows-style source path and disable path conversion:
  `MSYS_NO_PATHCONV=1 docker run --rm -v "C:/.../out:/work/out" cle-exp-towncrier ...`
  Prefer `make run`.
